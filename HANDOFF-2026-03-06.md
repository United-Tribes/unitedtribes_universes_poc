This is a handoff document for the Cast & Creators section of the Pluribus POC. It covers the full architecture of the cast detail page, how the broker API powers it, the three-tier modal system, the commerce/transaction layer, all work completed to date, and what remains open.

Where the comp and the running code differ, the code wins.

## The Commerce & Transaction Layer

**This is the thesis of the entire POC.** The transaction system proves that authorized content with real pricing creates a better experience than pirated links with no revenue path.

### The Commerce Thesis

The user journey is **DISCOVER → EXPLORE → ENGAGE → CONSUME → TRANSACT**. Every piece of content on the page has a revenue path. The POC demonstrates to investors that cross-media discovery is monetizable — not just engaging, but commercially viable.

### Transaction Types

**Primary consumption** (filled button, brand-colored background, white text):
The main free or included action — "▶ Watch on Apple TV+", "▶ Watch on YouTube", "Read", "Continue Reading · Subscribers". This is what the user is most likely to do.

**Purchase** (outlined button, white background, navy text, border):
Costs money — "Buy Season · $6.99", "Buy Episode · $0.99", "Purchase Article · $0.50". Visually subordinate to primary action. The user sees the free option first, purchase options second.

**Save to Library** (+):
The lightest transaction. Not revenue-generating itself, but expresses user intent and feeds the engagement funnel.

### Transaction Button Styling

```
Primary (consumption):
  background: [brand color — #000 for Apple TV+, #FF0000 for YouTube, #1a2744 for articles]
  color: white
  border: none
  font-weight: 700

Purchase (costs money):
  background: white
  color: #1a2744 (navy)
  border: 1.5px solid var(--border)
  font-weight: 600

+ Save:
  Default: white/transparent bg, border, "+" text
  Active: gold bg (#f5b800), gold border, navy text, "✓"
```

### Where Transactions Surface

**Card-level teases (Discovery Strip):**
- Pluribus card: `$0.99/ep · $6.99 season` (8px, textMid, fontWeight 600)
- NYT card: `Subscribers · Free`
- The Watch and Kimmel: no price tease (free YouTube content — the platform badge is sufficient context)

**Type 3 overlay modals (full transaction surface):**
- **Current build:** Platform pills, + Save button, and video player or article reader are implemented. The `priceTease` text is rendered on Discovery Strip cards (line 9341).
- **Design direction (not yet coded):** Each overlay should present the full transaction hierarchy — primary consumption button (filled, brand-colored), purchase buttons (outlined), and + Save. For example, Pluribus overlay: "▶ Watch on Apple TV+" (primary), "Buy Season · $6.99" (purchase), "Buy Episode · $0.99" (purchase). NYT overlay: "Continue Reading · Subscribers" (primary), "Purchase Article · $0.50" (purchase). This is the next priority for the commerce layer — the overlay is the consumption and transaction surface, and it's what investors see.

**Platform pills ("Also on:"):**
When content is available on multiple platforms, pills appear in the overlay showing each platform option. Each pill represents a different potential purchase/access path. The Watch shows Apple Podcasts, YouTube, and Spotify.

**KG Materials row:**
Every overlay modal has a "Related from the Knowledge Graph" section with 2-3 items. Each item has its own + button. These items are themselves content with transaction potential — clicking one could open its own overlay with its own transaction buttons.

**Relationship card media (Carol's World):**
Each expanded relationship card has media cards (e.g., "Helen's Flashback Scenes — Apple TV+", "Miriam Shor on Playing Helen — THR"). These carry platform badges and are themselves transaction opportunities.

**Cross-navigation links (Carol's World):**
The "Explore [Actor]'s universe →" links at the bottom of each relationship card are also a commerce surface — they navigate to another cast member's page, which has its own discovery strip, its own content cards, and its own transaction opportunities. Cross-navigation IS commerce. Every new page the user lands on has revenue paths.

### Future Commerce Integration Points

- Type 2 entity modals (when built) should surface transaction options for media referenced inside them
- Relationship card media items should eventually open Type 3 overlays with full transaction buttons
- Ask bar responses may reference content — entity names in responses that link to purchasable/streamable content represent organic transaction opportunities driven by the user's own questions

### Why This Matters for Investors

The POC proves three things:

1. **Authorized content creates better UX** — real platform badges, real prices, real embedded playback
2. **Cross-media discovery is monetizable** — every modal has transaction buttons with real pricing
3. **The knowledge graph makes everything connected** — KG materials in every modal mean every discovery leads to another transaction opportunity

The commerce layer isn't bolted on — it's woven into every interaction tier.

## The Three-Tier Modal System

### Type 1 — Entity Popover (Lightweight Float)

**Purpose:** Quick context on any entity mentioned in text. The user gets just enough to decide if they want to go deeper.

**Size:** 520px wide, auto height.

**Note on evolution:** The original comp design had Type 1 at ~340px with no search input — a pure lightweight tooltip. During the React build, Type 1 grew to 520px and absorbed a `PopoverAskInput` component, taking on some functionality that was originally intended for Type 2. This is why the Phase 2 spec describes a smaller, simpler popover — the build evolved Type 1 beyond the original design. This isn't a problem, but it means the visual and functional distinction between Type 1 and Type 2 needs careful design attention when Type 2 is built. They're similar width, so the escalation needs to feel different through depth of content, not just size.

**Trigger:** Click any `<EntityTag>` (blue underlined text) anywhere on the page — slug, bio, response text, article cards. Calls `openPopover(entityKey, event)` (line 12038) which captures the DOM bounding rect.

**Positioning:** Fixed position below the clicked element (`anchorRect.bottom + 8px`). Flips above if it would go off the bottom. Shifts left if off the right edge. Clamped to 12px from all viewport edges.

**Dismiss:** Escape key (immediate), click outside (50ms delay to avoid self-close from trigger), scroll 3+ times (200ms delay). Toggle: clicking the same entity again closes it.

**Contents:**
- 100×100px photo with type badge overlay (CHARACTER red, ACTOR navy, COMPOSER indigo, etc.)
- Name (18px/700), subtitle (12px)
- Bio (13px, 3-line clamp via WebkitLineClamp)
- Up to 2 KG videos from `quickViewGroups` or `interviews`
- `PopoverAskInput` component — text input with "Ask about [entity]..." placeholder and gold star icon. Queries pipe into the active broker thread.

**State:** `popoverEntity` (string) and `popoverAnchorRect` ({top, left, bottom, right, width, height}) at App level (line 11940).

**Render:** Portal to `document.body`. Animation: `popoverFadeIn 0.15s ease-out`. Z-index: 95.

**Special routing:**
- `_ep:*` keys navigate to EPISODE_DETAIL instead of showing popover
- `_song:*` keys show popover with embedded Spotify/YouTube player

### Type 2 — Entity Modal (Full Browsable Float) — Phase 2C, NOT YET BUILT

**Architectural decisions for the next developer:**

- **Single reusable component** — NOT pre-built divs for each entity. One `EntityModal` component that accepts an entity key, looks up KG data, and renders dynamically.
- **Floats on top of the page** anchored near the clicked text. `position: absolute` or `position: fixed`, z-indexed above page content but below Type 3 overlay. Does NOT push content down or displace anything. Does NOT dim the page or lock scroll.
- **Sizing:** ~500-600px wide, height determined by content. Larger than Type 1 but doesn't take over the viewport.
- **Contains:** Name, subtitle, narrative paragraph, action buttons, KG materials row with individual + buttons, search input with search pills, + save button. Can reference video and media content but does NOT play them — video playback escalates to Type 3.
- **Trigger:** User clicks an action button inside a Type 1 popover (e.g., "Full Universe" or "5 Interviews"). Type 1 closes, Type 2 opens in approximately the same area.
- **Dismiss:** × button, click outside, opening a different entity's modal.
- **Data shape:** Consumes the same entity data from `pluribus-universe.json` — `{title, subtitle, description, type, photoUrl, bio[], completeWorks[], collaborators[], quickViewGroups[], interviews[], character_profile}`. Renders sections based on what data exists for that entity.

**Key distinction from Type 1:** Type 1 is quick context (520px with PopoverAskInput). Type 2 is a full browsable exploration surface with KG materials, multiple action paths, and search. Both float. Neither disrupts the page. Because the current Type 1 has grown beyond the original comp's lightweight design, the visual and functional distinction between Type 1 and Type 2 needs careful design attention when Type 2 is built — they should feel like clearly different tiers, not slight variations of the same thing.

**Key distinction from Type 3:** Type 2 floats near where the user clicked and doesn't dim the page or lock scroll. Type 3 takes over the entire viewport and is specifically for media consumption.

### Type 3 — Discovery Overlay (Full-Screen Player/Reader)

**Purpose:** Media consumption — watching video or reading an article. The user has committed to engaging with this content.

**Size:** Max 620px wide, max 85vh tall, centered on dark blurred backdrop (`rgba(10,14,26,.75)` + 6px blur).

**Trigger:** Click a Discovery Strip card or article card. Sets `discoveryOverlay = {id, data}`.

**Dismiss:** Click backdrop, Escape key, or close button (32px circle, top-right).

**Two template variants:**

**Video Player:**
- 16:9 black area with gradient background from card data
- Frosted glass play button (68px circle, `rgba(255,255,255,.12)` + 8px blur + 2px white border, scales to 1.06 on hover)
- Platform badge (bottom-left, brand-colored pill)
- Duration (bottom-right, 9px, semi-transparent background)

**Article Reader:**
- Off-white area (`#faf8f5`)
- Publication name (8px, uppercase)
- Headline (22px, Georgia serif, 800-weight, navy)
- + Save button inline with headline
- Byline (11px)
- Excerpt (13px, Georgia serif, 1.65 line-height)
- "Continue reading →" gold text (styled as a link but currently non-functional — no onClick handler attached. Placeholder for future in-platform expanded reader, NOT an external link)

**KG Materials Row:** "Related from the Knowledge Graph" header with gold bar accent. Up to 3 cards, each with color swatch + ▶ icon, title (10px/700), meta (8px), individual + save button. Border turns gold when saved.

**State:** `discoveryOverlay` (null or {id, data}), `overlaySaved` (boolean), `kgSaved` ({index: boolean}).

### Escalation Roadmap (Type 1 → Type 2 → Type 3)

**CURRENT STATE:**

```
Entity link click → Type 1 popover (independent)
Discovery strip card click → Type 3 overlay (independent)
No connection between the two systems.
```

**INTENDED ARCHITECTURE:**

```
Entity link click (anywhere on page)
  └── Type 1 popover opens (small float, anchored to clicked text)
       └── User clicks action button inside popover
            └── Type 1 closes → Type 2 opens (large float, same area)
                 └── User clicks video/media item inside Type 2
                      └── Type 3 player overlay opens (full screen)

Discovery strip card click
  └── Type 3 player overlay opens directly (no intermediate steps)

Closing reverses the chain:
  Close Type 3 → Type 2 still visible underneath
  Close Type 2 → back to reading, page never moved
```

Each step represents deeper user commitment. The system rewards that commitment with more content and more interaction options. The user can stop at any tier.

**Implementation consideration:** The "Close Type 3 → Type 2 still visible" behavior requires intentional z-index management. Type 3's backdrop (z-index: 200) must not destroy Type 2's state. Type 2 should sit at a z-index above page content (~100-150) but below Type 3's backdrop.

## + Button / Library System

This system touches every component on the page.

### Where + Buttons Appear

- Discovery Strip cards (bottom-right of each card)
- Type 3 overlay modals (in the title area, inline with headline)
- KG materials items inside overlay modals (each item has its own +)
- Relationship card media items (within Carol's World expanded cards)
- Anywhere `PlusBtn` component is rendered (line 8660)

### Toggle Behavior

- **Default state:** "+" text, white/transparent background, tan border (`#d8cfc2`), textDim color
- **Saved state:** "✓" text, gold background (`#f5b800`), gold border, navy text
- **Click toggles** between states
- All + buttons write to the same `library` Set (persisted to `localStorage` as JSON array)
- `libCount` tracks session additions from cast detail for potential badge display
- `PlusBtn` component (line 8660) checks `library.has(itemTitle)` for current state

### Follow vs Save — Critical Distinction

**+ Follow** (hero area, next to social pills) = **Relationship.** This means "I care about this person — send me new content about them over time." It's a subscription. Engagement metric. Click expands to show Instagram/YouTube/TikTok social pills (individually toggleable). Click again to unfollow and collapse.

**+ Save** (everywhere else) = **Static bookmark.** One-time save of a specific piece of content to the user's library. Content metric.

These are fundamentally different operations in the data model. Follow creates an ongoing relationship. Save creates a point-in-time bookmark. The UI should never conflate them.

**Current implementation:** Both write to the same `library` Set via `toggleLibrary()` — the Follow button at line 8629 calls `toggleLibrary(selectedPerson)` identically to how + Save works. **In production, these would be separate systems** with different backend calls, different UI feedback, and different data implications.

## Entity Linking System

Two components power entity linking across the entire page:

### `EntityTag` Component (line 588)

A `<span>` wrapper for clickable entity names:
- **Clickable** (has `onClick`): Blue text (`T.blue` = `#2563eb`), underline, fontWeight 600, pointer cursor. **Note:** The cast detail page defines its own `C.link` = `#1565c0` (a darker, less saturated blue). EntityTag uses the global `T.blue` value regardless of which screen it's on — the two blues are different hex values.
- **Not in KG** (no `onClick`): Warm muted text, default cursor, tooltip: "Entity not yet available in knowledge graph"

### `linkEntities()` Function (line 612)

Scans text and auto-wraps entity names in `<EntityTag>` components.

**How it works:**
1. Takes a pre-sorted entity name list (longest-first, built at line 12034)
2. For each name, builds a word-boundary regex (`\bName\b`, case-insensitive `gi` flag)
3. Splits text into match/non-match chunks, wraps matches in `<EntityTag>` with onClick handlers
4. Resolves aliases (e.g., "Body Snatchers" → "Invasion of the Body Snatchers (1956)")
5. Only creates clickable tag if entity exists in KG `entities` object

**Exclusions:**
- `ENTITY_LINK_EXCLUDE`: `["BTR1", "Ricky Cook"]` — never linkable
- `COMMON_WORD_EXCLUDE`: ~60 English words (her, him, the, and, etc.)
- Names under 4 characters excluded

**Special entity types:**
- Episode titles (`_ep:S1E1`) → navigate to EPISODE_DETAIL screen instead of popover
- Song titles (`_song:3`) → show popover with Spotify/YouTube player embed

### 4. Progressive Disclosure Flow

When a path chip is clicked:

1. **Bio loads** from broker API (KG-grounded prompt) — `BioThinkingPill` shows with cycling step text + gold strobe
2. **Search chips appear** (3 starter questions) — staggered fade-in (`flowIn 0.4s ease 0.15s`)
3. **Ask bar appears** — text input for conversational follow-ups (`flowIn 0.4s ease 0.25s`)
4. **"Discover [Name]'s World" chip** appears below — gates deeper sections (`flowIn 0.3s`)
5. **Click Discover World** → deeper sections expand at once. **Actor path:** articles, work, Go Deeper (videos/podcasts). **Character path:** relationship cards, Deep Dive, Sonic Connection

The "Discover World" chip is a toggle — click again to collapse. Visual feedback: active cream background, gold border, gold glow shadow, arrow rotates 90°. All content below loads when the boolean flips, though broker API section summaries fire lazily on first render.

**Note:** The "Discover World" progressive disclosure gate was added during the React build — it doesn't exist in the original comp, where all path sections were visible immediately when a path opened. It was added to manage visual load and create a natural loading boundary for API calls.

## Path Collapse/Expand with State Retention

### Current Behavior

- Clicking a path chip sets `activePath` to that path and adds it to `visitedPaths` Set
- Switching paths changes `activePath` — the previous path's content is hidden but not destroyed
- **State is retained per-path:** `pathBios`, `pathAskConvo`, `pathAskInput`, and `worldExpanded` are all keyed by path (`{rhea: ..., carol: ...}`) and persist across switches
- **Conversation threads survive path switching** — if you asked 3 questions on Carol's path, switched to Rhea, then came back to Carol, all 3 Q&As are still there
- **Expanded relationship cards persist** — if Carol & Helen was expanded, it's still expanded when you return
- **Per-path conversation state also persists when the user collapses and re-expands the "Discover World" gate** — the `worldExpanded` toggle doesn't affect `pathAskConvo`
- Threads are **only cleared** when navigating to a different cast member entirely (the `selectedPerson` change useEffect resets everything)

### Known Quirk: `activeDeeper` Not Keyed by Path

`activeDeeper` is a single string value (e.g., `"d-bcs"` or `"d-immune"`), not keyed by path. It accidentally works because Rhea and Carol deeper block IDs don't collide — Rhea uses `d-bcs`, `d-globe`, `d-cast` while Carol uses `d-immune`, `d-joining`, `d-ep7`. When you switch paths, the stored ID doesn't match any deeper block on the other path, so nothing shows. When you switch back, it still matches, so it re-renders. **Should be properly keyed by path if more paths are added.**

### visitedPaths Visual Indicator — OPEN ITEM

The `visitedPaths` Set exists and updates correctly on every path open. **No visual indicator is currently rendered** on previously-visited path chips. The decided design: a subtle visual cue (gold bottom border or slightly different background) on path chips the user has already visited, signaling "you've been here, click to reopen." This is not wired yet — `activePath` drives chip styling, but `visitedPaths` does not.

## Broker API Integration

### Endpoint

```
POST https://166ws8jk15.execute-api.us-east-1.amazonaws.com/prod/v2/broker
Content-Type: application/json
Body: { "query": "<prompt>" }
Response: { "narrative": "<text>", ... }
```

### Bio Caching (localStorage)

Bios are cached to avoid redundant API calls. Three helper functions (`getBioCache`, `setBioCache`, `clearBioCache`) manage `localStorage` keys:

```
Key:       ut_bio_[name]  (name lowercased, spaces → underscores)
Value:     { narrative: string, timestamp: number, edited: boolean }
```

- **On path open:** checks cache first. If a cached bio exists, it's used immediately without an API call.
- **`edited` flag:** If true, the bio was manually edited by a user and should NOT be overwritten by a fresh API call.
- **State vars:** `bioCacheTimestamp` and `bioCacheEdited` track the current bio's cache metadata.
- **Cache clear:** `clearBioCache(name)` removes the key entirely, forcing a fresh API fetch on next open.

### Where the API Is Called

| Purpose | Trigger | Prompt Summary |
|---------|---------|----------------|
| **Path bio** | Path chip opens (cache miss) | 3-4 sentence bio, KG-grounded, mentions relationships |
| **Live bio** (lobby) | Cast member selected | `buildKGBioPrompt()` — type-aware bio generation |
| **Ask bar** | User types question | Full conversation context + KG + FOLLOW_UPS instruction |
| **Article summary** | Discover World expands | 1-2 sentence media coverage overview |
| **Work summary** | Discover World expands | 1-2 sentence filmography overview |
| **Crew bio** | Crew member selected | Same as live bio but for crew |

```
[5. USER QUESTION]

These must be 3 different follow-up questions a curious person would naturally ask
next, based on your answer. Separate with | characters. Keep each under 8 words.
Do NOT skip this line.
```

### FOLLOW_UPS Parsing (line 9048)

The broker returns something like:
```
Carol's immunity stems from profound grief over her late wife Helen...

FOLLOW_UPS: What happened to Helen? | Are other immune people grieving? | Does Carol's immunity weaken?
```

**Parsing logic:**
1. Case-insensitive search for `FOLLOW_UPS:` marker
2. Extract everything after the marker, strip it from the narrative
3. Try pipe-split first (`|`) — if 2+ items, use those
4. Fallback: split by newlines, strip leading numbers/dashes/bullets (`/^\s*[\d\-\.\•\*]+[\.\):]?\s*/`)
5. Filter: keep only items 3-60 chars, take first 3
6. **If <2 found:** Generate fallback questions from entity names in the response: `"Tell me more about ${entityName}?"`

### Conversation History Accumulation

- **Per-path isolation:** `pathAskConvo = { rhea: [], carol: [] }` — completely separate threads
- **Sliding window:** Only last 2 completed exchanges sent as context, each response truncated to 300 chars
- **Growth:** Each question appends to the array. No limit on thread length (only the context window slides)
- **Persistence:** Thread survives collapse/re-expand of "Discover World" and path switching. Cleared only when navigating to a different cast member
- **Response data:** Each conversation entry stores `{query, response, loading, error, followUps, responseEntities}`

### Section Summary Prompts

**Articles:** `"In 1-2 sentences, summarize the media conversation around [actor] right now. Key coverage includes: [article titles and mastheads]. Be specific and editorial — this is a brief overview for an articles section. Do not use generic phrases."`

**Work:** `"In 1-2 sentences, summarize [actor]'s career and body of work. Key credits include: [top 12 works with year and role]. Be specific and editorial — this is a brief overview for a filmography section. Do not use the word 'boasts' or generic phrases."`
